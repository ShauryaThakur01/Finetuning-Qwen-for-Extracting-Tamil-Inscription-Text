# Fine-Tuning Qwen2.5-VL for Ancient Tamil Inscription

## Abstract

Ancient stone inscriptions written in scripts such as Tamili and Vattezhuthu are an important source of historical and linguistic knowledge. However, reading and interpreting these inscriptions is difficult because the text is often damaged, unevenly carved on stone surfaces, and written in scripts that are no longer commonly used. In addition, there are very limited labeled datasets available for training modern machine learning models.

This project explores how recent vision–language models can be used to automatically recognize and interpret these inscriptions. The proposed system combines the transformer-based vision language model **Qwen 2.5-3B** to extract text from images of inscriptions. Because real inscription datasets are limited, synthetic images are generated by rendering text and placing it on stone-like backgrounds. We utilized the **Aksharamukha** Python package to convert inscription text to images and the **Augraphy** library along with standard pre-processing to blend the inscription image onto a stone background.

The Qwen 2.5-3B model is then fine-tuned using the parameter-efficient method **LoRA** on our custom dataset, and a **Qwen 2.5-0.5B** model is used to convert the recognized characters to human-readable form. The system is evaluated using **BLEU** and **ROUGE** metrics. The overall goal is to investigate how modern NLP techniques can support the digital preservation and interpretation of historical inscriptions.

## Introduction

Ancient stone inscriptions, particularly those written in archaic scripts like Vattezhuthu and Tamili, serve as the primary source of socio-political and linguistic evolution of South Asia. These epigraphic artifacts are foundational to understanding the transition of languages like Tamil — one of the oldest languages in the world — from ancient roots to modern forms. However, archaeologists face challenges such as physical degradation of stone inscriptions due to centuries of erosion, and a declining number of paleography experts capable of deciphering these inscriptions.

To tackle these challenges, this research investigates the use of **Vision–Language Models (VLMs)** for the automatic recognition and translation of ancient inscriptions. We propose a pipeline based on the **Qwen 2.5-3B** architecture, a transformer-based model capable of processing both visual and textual information.

A synthetic dataset generation framework is developed to address the limited availability of labeled datasets. The inscription text is first rendered using the **Aksharamukha** Python package, which converts modern Tamil script into its ancient predecessors (Tamili and Vatteluttu). The **Augraphy** library simulates realistic stone textures, lighting variations, and surface degradation to produce training images that closely resemble real-world epigraphic images.

The core methodology focuses on fine-tuning the Qwen model using **Parameter-Efficient Fine-Tuning (PEFT)** with the **LoRA (Low-Rank Adaptation)** technique. This allows the model to learn the characteristics of Tamili and Vattezhuthu scripts without requiring the full computational cost of updating all model parameters. A **Qwen 2.5-0.5B** model is then used to convert the learned words to human-readable form.

## Dataset

To fine-tune the Qwen 2.5-3B and Qwen 2.5-0.5B models, a high-quality, domain-specific dataset was required. Since no publicly available dataset exists, the first task was to create paired image-text data containing:
- An image paired with inscription text (for fine-tuning Qwen 2.5-3B)
- The corresponding original Tamil text (for fine-tuning Qwen 2.5-0.5B)

**Pipeline steps:**

1. **Collecting Tamil text data** — Foundational Tamil text was sourced from the open-source dataset **Project Madurai**, which provides Tamil Unicode UTF-8 text. Raw continuous strings were pre-processed and segmented into meaningful, uniform chunks.

2. **Converting to ancient scripts** — A custom typographical pipeline automated the conversion of modern Tamil text chunks to their ancient script equivalents (Tamili and Vattezhuthu), using the **Aksharamukha** Python library to convert these chunks into text images.

3. **Combining text and stone background** — The **Pillow** Python library was used to blend the text images onto stone background images sourced from open-source image websites. The **Augraphy** Python library was then applied to add realistic degradation effects.

4. **Final dataset** — The pipeline produced approximately **28,000 paired text–image samples**. Ground-truth data was converted into a conversational **JSONL** structure compatible with the Qwen2.5-VL architecture, explicitly linking each simulated stone image with its corresponding modern Tamil transcription. Due to hardware restrictions, a sample of **5,000 image-text pairs** was used for training.

## Methodology

To overcome the dual challenges of severe epigraphic degradation and continuous-script morphology, a **decoupled, two-stage cascaded architecture** was implemented. A monolithic approach — forcing a single model to perform both visual extraction and linguistic Sandhi (*Puṇarcchi*) resolution — yields sub-optimal convergence and exceeds the 20GB VRAM hardware constraint.

### Stage 1: Visual Extraction Expert

The **Qwen2.5-VL-3B-Instruct** model was utilized, leveraging its native **Multimodal Rotary Positional Encoding (mRoPE)** to process uneven stone surface topography without aspect-ratio distortion.

Since the base tokenizer lacked extinct Tamili and Vattezhuthu characters, a **vocabulary injection protocol** was engineered: missing tokens were extracted from a custom tokenizer and injected into the 3B model by dynamically resizing the GPU embedding matrix.

**Training configuration:**
- Quantization: **4-bit NormalFloat (NF4) QLoRA**
- LoRA adapters: `r=16`, `α=32`, targeting attention heads
- Core MLP layers frozen to prevent catastrophic forgetting
- `embed_tokens` and `lm_head` layers explicitly unfrozen to learn the new vocabulary
- Optimizer: `paged_adamw_8bit` with expandable memory segments
- Dataset: **6,000 synthetic image-text pairs** (80/20 train/test split), stratified across Tamili and Vattezhuthu classes

### Stage 2: Morphological Segmentation Expert

The continuous string output from Stage 1 is passed to **Qwen2.5-0.5B**, an ultra-lightweight causal language model dedicated exclusively to resolving grammatical Sandhi boundaries and inserting modern spacing.

A **sliding-window chunking algorithm** was engineered across the text corpora, with inputs artificially restricted to **15-word micro-prompts**. This generated **25,611 highly localized training pairs** (1,000 reserved for validation). The constrained context window forces the model's attention to focus on local syntax rather than hallucinating overarching semantic narratives, resulting in a highly responsive, low-latency deployment.

## Results & Evaluation

Two evaluation metrics were used: **BLEU** and **ROUGE**.
- **BLEU** measures how accurately the model extracts text from a given image.
- **ROUGE** scores measure how accurately the model recognizes each character.

### Vision Model (Qwen 2.5-3B) — Clean Images

| Metric   | Score  |
|----------|--------|
| BLEU     | 95.40  |
| ROUGE-1  | 0.9881 |
| ROUGE-2  | 0.9657 |
| ROUGE-L  | 0.9838 |

### Vision Model (Qwen 2.5-3B) — Noisy Stone Images (Test Set)

| Metric   | Score  |
|----------|--------|
| BLEU     | 63.02  |
| ROUGE-1  | 0.8796 |
| ROUGE-2  | 0.7087 |
| ROUGE-L  | 0.8032 |

### Segmentation Model (Qwen 2.5-0.5B)

| Metric   | Score  |
|----------|--------|
| BLEU     | 72.02  |
| ROUGE-1  | 0.8471 |
| ROUGE-2  | 0.7636 |
| ROUGE-L  | 0.8469 |

The high scores on clean images confirm that Qwen 2.5-3B was trained effectively on the training set. The noisy stone image results demonstrate solid performance on realistic epigraphic inputs. The segmentation model shows strong capability in converting continuous ancient script output into human-readable modern Tamil.

## Future Work

Although the system shows promising results for recognizing ancient Tamil scripts, several improvements remain:

- Train with **larger datasets** to improve generalization
- Explore a **unified multimodal model** for direct image-to-text transcription, eliminating the two-stage pipeline
- Extend the pipeline to **other historical scripts** such as Grantha, Brahmi, and early Kannada
- Evaluate **larger vision–language models** such as Qwen3-VL (e.g., Qwen3-VL-235B-A22B-Instruct)

## My Experience

### Relevance of Course Content

Several foundational concepts were essential for this project. **Unit 2's coverage of Transformer-based architectures** helped interpret various vision-language models and informed the selection of Qwen 2.5-3B. The **lab experience with task-specific methods** was useful for discovering Python packages like Aksharamukha, Pillow, and Augraphy. **Section 3.2 from Unit 03** on Language Filtering and Deduplication helped curate high-quality text data for the model.

### Tokenization Fairness

During fine-tuning, unfair tokenization was encountered because the base model lacked specific ancient Tamil characters — a problem previewed in the course's first lab, which demonstrated how character-per-token ratios differ for non-English languages. This was solved by extracting Tamil tokens from a custom tokenizer and injecting them into the model.

### Two-Stage Architecture

The two-stage pipeline was inspired by the **Mixture of Experts** concept from Unit 03. Given hardware and memory constraints (20GB VRAM), separating the OCR and segmentation tasks into specialized models proved more effective than a single monolithic approach.

### Reflections

Given what was learned, future work would prioritize:
- Assessing hardware and memory requirements *before* scaling the dataset
- Exploring additional VLMs beyond Qwen 2.5-VL
- Building more balanced datasets from the start

The course content was sufficient as a foundation, though additional background in vision-language model fine-tuning and balanced dataset construction would have reduced implementation overhead within the five-week timeframe.

## Reflective Analysis of Learning Outcomes

### What Did You Learn?

This project demonstrated that **a model's performance is often a reflection of how well you respect the constraints of data and hardware**. Finetuning is not a brute-force process — architecture matters. No amount of training data is useful if the model's internal vocabulary (the tokenizer) cannot "see" the characters correctly. The decision to split the task into two models taught the value of **architectural humility**: better performance often comes from structural constraints rather than raw parameter count.

### How Was It Learned?

The monolithic model approach initially produced hallucinated character sequences. Tracing this back to the Tokenization Fairness concept from the course revealed the root cause: the base model treated many ancient characters as unknown or fragmented sub-words. This required manually extracting tokens from a custom tokenizer and dynamically resizing the GPU embedding matrix — manipulating the model's "sight" rather than just its "memory."

The insight into architectural humility came directly from the 20GB VRAM hardware constraint. Attempting to handle both OCR and segmentation in a single model led to sub-optimal convergence, motivating the decoupled pipeline design inspired by Mixture of Experts theory.

### How Did the Project Support the Main Learning Objective?

The project supported the objective of being able to **seek, assess, and use scientific information within the area of NLP**:
- **Sought** the Qwen 2.5-VL technical report to understand mRoPE for processing uneven stone surfaces
- **Assessed** the PEFT/LoRA literature to determine how unfreezing `embed_tokens` and `lm_head` was the key to fixing unfair tokenization of ancient scripts
- **Applied** the mixture-of-models technique to architect a two-stage pipeline, solving the 20GB VRAM bottleneck

### Broader Impact

Researchers and professionals can learn that **synthetic data generation pipelines** can effectively tackle data unavailability in low-resource domains, and that **decoupling complex tasks into multi-stage pipelines** can yield significantly better results than monolithic approaches — especially under hardware constraints.
